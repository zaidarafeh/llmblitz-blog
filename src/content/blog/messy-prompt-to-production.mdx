Most prompts start the same way: someone writes a quick instruction, it mostly works, they ship it. Three months later it's causing incidents and nobody remembers why it was written that way.

This is the process for doing it properly — from a rough first draft to something you'd put in production with confidence.

## The starting prompt

This is a real prompt (anonymized) from a team building an AI writing assistant:

```
You are a helpful writing assistant. Help the user improve their text. 
Make it better, clearer, and more professional. Consider the context and 
what the user is trying to achieve. Be helpful and provide good suggestions.
```

It "works" in the sense that it produces improvements. But it fails in three ways they hadn't noticed yet:
1. It produces wildly inconsistent output length
2. It occasionally rewrites the user's voice out of the text entirely
3. It sometimes adds unsolicited structural changes (headers, bullets)

Let's trace exactly why.

## Step 1: Diagnose with token analysis

The first thing we do is run this through BlitzLab's single analysis mode with a few real inputs.

{/* SCREENSHOT: BlitzLab output — token confidence heatmap on the writing assistant prompt */}

**What BlitzLab found:**

The finding on the first run: *"The phrase 'Make it better, clearer, and more professional' caused a 38% confidence drop on the first output token across all 5 test inputs — the model was uncertain whether to begin with the edited text, an explanation of changes, or a suggestion."*

That's the core problem. Three adjectives ("better", "clearer", "more professional") that mean different things are forcing the model to resolve an ambiguity before it can start generating. It's picking randomly between output strategies.

The recommendation: *"Replace 'Make it better, clearer, and more professional' with a single explicit instruction specifying what the output should contain."*

## Step 2: Rewrite with constraints

Based on the diagnosis, here's the rewrite:

```
You are a writing editor. Rewrite the text below to improve clarity and concision 
while preserving the author's voice and original structure.

Rules:
- Do not add headers, bullets, or sections that weren't in the original
- Do not change the tone — match the formality level of the input
- Output only the rewritten text, no explanation or commentary
- If the text is already clear and concise, return it unchanged
```

Four changes:
1. Replaced vague adjectives with two specific goals ("clarity and concision")
2. Added explicit constraints on what *not* to do (the root cause of the voice and structure issues)
3. Added output format constraint ("only the rewritten text")
4. Added an edge case handler for already-good text

## Step 3: Test the rewrite empirically

We take the rewritten prompt to Prompt Designer, provide 5 real input examples with expected outputs, and run it against the same model.

{/* SCREENSHOT: Prompt Designer results — ranked candidates with confidence scores */}

The rewritten prompt scored **84/100** on empirical testing vs **51/100** for the original. More importantly, output variance dropped — the distribution of outputs tightened significantly around what the team actually wanted.

We also tested two variants generated by Prompt Designer:
- One that added an XML structure for the output
- One that used a role definition with explicit constraints

The XML variant scored highest (91/100) but the team decided the simpler rewrite was sufficient for their use case. Don't over-engineer past your actual requirements.

## Step 4: Optimize for cost

Before shipping, we ran both the original and rewritten prompt through EcoBlitz at the team's actual volume: ~15,000 calls/month.

{/* SCREENSHOT: EcoBlitz results — original vs rewritten prompt cost comparison */}

| | Original | Rewritten | Savings |
|--|----------|-----------|---------|
| Avg input tokens | 285 | 210 | −26% |
| Avg output tokens | 340 | 180 | −47% |
| Cost/month @ 15k calls | $3.12 | $1.44 | **−54%** |

The output token reduction was the big one — the original's vague instructions caused the model to generate longer, more exploratory outputs before settling. The rewritten prompt committed immediately.

**$3.12 → $1.44/month** doesn't sound dramatic. But this team had 4 similar prompts running at the same volume. The total saving across all four: **$25.20/month**, without touching a single line of application code.

## The production checklist

Before you ship any LLM prompt, run through this:

**1. Does it specify what the output *contains*?**
Not "be helpful" — but "return X in format Y".

**2. Does it constrain what the output *doesn't* contain?**
If you don't want headers, say so. If you don't want explanations, say "no explanation".

**3. Have you tested it on your actual inputs, not just obvious ones?**
Edge cases surface in production. Test with your weirdest real inputs first.

**4. Do you know how confident the model is across those inputs?**
Low-confidence tokens on early positions mean the model is deciding between output strategies on each call. That's where inconsistency comes from.

**5. Do you know what it costs at your real volume?**
A prompt that costs $0.0003/call is $300/month at 1M calls. Know this before you ship.
