**TL;DR**: Rewriting a prompt cut its monthly LLM cost by 80% and raised its quality score from 78% to 98%, but the cost-optimized version's token confidence dropped by 10 points. Here's a 4-step process we used, and how to decide if the tradeoff is worth making for your use case.

Most prompts begin the same way: someone writes it as an instruction, it mostly works, they ship. Two weeks later it's causing incidents.

Here are the four steps I use to take a prompt from 'kind of works' to 'ship it'.

## The starting prompt

This is a real prompt (anonymized) from a team building an LLM-based writing assistant:

```plain
You are a helpful writing assistant. Help the user improve their text. 
Make it better, clearer, and more professional. Consider the context and 
what the user is trying to achieve. Be helpful and provide good suggestions.
```

This prompt "works" in the sense that it makes some improvements. But it fails in 3 ways they hadn't noticed yet:

1. It produces wildly inconsistent output length
2. It occasionally rewrites the user's voice out of the text entirely
3. It sometimes adds unsolicited structural changes (headers, bullets)

Let's trace exactly why.

## Step 1: Diagnose with token analysis

The first thing we do is run this through BlitzLab (a prompt analysis [tool](https://llmblitz.io/)) with a few real example inputs. For input text, I will use the opening statement from [Steve Jobs' Stanford speech](https://news.stanford.edu/stories/2005/06/youve-got-find-love-jobs-says).

**What BlitzLab found:**

![Blitzlab Token Analysis Portion](/images/Tokens.png "Blitzlab Token Analysis Portion")

The finding on the first run: _"The prompt's instruction to 'Make it better, clearer, and more professional' caused the model to systematically replace the user's (Steve Jobs') casual phrasing  with formal alternatives, creating high uncertainty on tokens where informality and formality compete—ex: token 83 '**meaningful**' (2% confidence, with '**simple**' at 30%) show **the model torn between matching the user's conversational tone and elevating to professional language**." This is the stuff that creates variation in LLM output._

**Ex: the LLM was \~2% confident about picking the word 'meaningful'**

![The LLM was ~2% confident about picking the word 'meaningful'](/images/Tokens3.png "The LLM was ~2% confident about picking the word 'meaningful'")

According to BlitzLab's analysis report, this is the core  problem: "**The word 'professional' signaled the model (GPT-4o mini) to formalize everything, so it kept second-guessing whether to preserve 'no big deal' or upgrade it to corporate language—the low confidence scores show it was genuinely uncertain about whether you wanted the casual tone kept**."

**BlitzLab recommendation**: _"Replace 'Make it better, clearer, and more professional' with 'Make it clearer and more polished while keeping the author's original voice and tone.' This tells the model to improve without erasing the casual, personal style that makes the opening effective."_

## Step 2: Rewrite with constraints

Based on the diagnosis, we need to rewrite the prompt with more specific instructions that preserve the essence of the input text and with the addition of constraints. (you can grab simple and free templates for writing prompts in BlitzLab's [Template Gallery](https://llmblitz.io/optimize?mode=compare&gallery=1)). Here is a sample I picked up from the gallery that you can use:

```plain
## Role:
// Describe who the model should act as (e.g. "You are a customer support agent for a SaaS company")

## Task:
// Describe what the model should do in one clear sentence (e.g. "Classify the customer message below into one of the categories listed")

## Rules:
// Add any constraints or formatting rules, one per line (e.g. "Output only the category name, no explanation")
// (e.g. "If the input is ambiguous, ask for clarification")

## Output format:
// Describe exactly what the response should look like (e.g. "A single word: Billing, Technical, or General")
```

Alternatively, we will use a [tool](https://llmblitz.io/prompt-surgeon) called Prompt Designer to improve the prompt we tested in a more systematic and empirical way, leading to even better working prompt. Here is the improved prompt generated by Prompt Designer

```plain
You are a writing assistant. Improve the user's text to make it clearer and more natural while keeping their voice and intent intact. Output only the revised text — no explanations unless asked.

Here are examples of the kind of editing you should do:

**Example A — Personal narrative:**
Original: "The first story is about connecting the dots. I dropped out of Reed College after the first six months, but then stayed around as a drop-in for another 18 months or so before I really quit."
Revised: "The first story is about connecting the dots. I dropped out of Reed College after six months, but I stayed on as a drop-in for another 18 months before I finally left."
→ Tightened phrasing, removed filler words, kept the conversational tone.

**Example B — Professional communication:**
Original: "We are implementing a new project management system starting next month. The reason for this change is that our current system has become outdated and is not meeting our needs anymore."
Revised: "Starting next month, we'll be rolling out a new project management system. Our current one is outdated and no longer meeting our needs, so this change will help us work more efficiently."
→ Reordered for impact, made tone warmer and less corporate, cut redundancy.

Apply this same approach to any text the user provides: tighten, clarify, and polish without changing meaning or adding new content.
```

Changes Made:

1. The instructions and the description of changes are a lot more specific
2. The new prompt preserves the tone/style of the original text
3. The new prompt provides examples to help the LLM understand what "good" output looks like

## Step 3: Test the rewrite empirically

We provide 2 real input examples with expected outputs in Prompt Designer, and run it against the same model.

![](/images/PromptDesigner.png)

The rewritten prompt scored **98/100** on empirical testing vs **78/100** for the original. More importantly, output variance dropped — the distribution of outputs tightened significantly around what the team actually wanted.

We also tested two variants generated by Prompt Designer:

- One that added an XML structure for the output
- One that used a role definition with explicit constraints

The XML variant scored (91/100) but the team decided the simpler rewrite was sufficient for their use case. Don't over-engineer past your actual requirements.

## Step 4: Optimize for cost

Before shipping, we wanted to estimate the monthly cost of running our prompt. We ran both the original and rewritten prompt through a [tool](https://llmblitz.io/eco-blitz) called EcoBlitz at the team's volume: \~15,000 calls/month. This is a modest daily volume, but as you can imagine, as daily calls get higher for typical tools, cost gets higher too. We also used the cost reduction feature in EcoBlitz to see if we can reduce cost from our prompt without compromising too much on efficacy.

![](/images/EcoBlitz.png)

**The result**: We were able to achieve **80-81% cost savings** by rewriting the prompt to optimize for token reduction. The cost came in the form of model confidence: BlitzLab's average token confidence (a different metric than the Prompt Designer empirical score above) dropped from 98% to 88%. This is a tradeoff that might or might not be worth making, and the best way to tell is to retest the low cost prompt in BlitzLab and see how consistent its responses are. We did that test, and the tradeoff turned out to be worth it, because the prompt produced sufficiently consistent results.

EcoBlitz reduces LLM cost in a numbers of ways. The most straightforward way is including token count restrictions in the prompt (something you can easily add to your prompt). The original prompt did not have a length anchor, so the model defaulted to \~400-token replies. The rewrite says 'Keep your response between 50 and 200 tokens', collapsing the number of tokens by more than half. A variety of other techniques were used as well.

## The production checklist

Before you ship any LLM prompt, run through this:

**1. Does it specify what the output _contains_?**
Not "be helpful" — but "return X in format Y".

**2. Does it constrain what the output _doesn't_ contain?**
If you don't want headers, say so. If you don't want explanations, say "no explanation".

**3. Have you tested it on your actual inputs, not just obvious ones?**
Edge cases surface in production. Test with your weirdest real inputs first.

**4. Do you know how confident the model is across those inputs?**
Low-confidence tokens on early positions mean the model is deciding between output strategies on each call. That's where inconsistency comes from.

**5. Do you know what it costs at your real volume?**
A prompt that costs $0.003/call is $3000/month at 1M calls. Know this before you ship.
